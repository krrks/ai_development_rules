# AI_DEVELOPMENT_RULES.md

**Universal AI Collaboration Development Rules (for any GitHub project)**

**Version**: 1.0

**Target audience**: Projects using Claude / Grok / Cursor / GitHub Copilot or any other AI coding assistant

**Core goal**: Enable AI to contribute code safely, efficiently, and incrementally while keeping the repository clean, CI-automated, and fully traceable.

---

## 1. Purpose & Core Principles

These rules establish a standardized **"Human oversight + AI incremental contribution"** workflow.

AI never edits the repository directly. Instead, it submits changes as **structured Patch ZIP files** that are automatically validated and applied by CI.

**Core Principles**:

- **Minimal changes only**: Modify only what is necessary. Keep changes small and focused.

- **Automation first**: Let GitHub Actions handle all repetitive work.

- **Human final control**: AI must never touch .github/, CI config files, or core root files.

- **Fully auditable**: Every change is logged (Patch + CHANGELOG).

---

## 2. Required Project Infrastructure (Repository Setup)

The following structure **must exist** in the project root (CI will maintain it automatically):


.

├── zip\_update/

│   ├── patches/          # AI-generated patch ZIPs are placed here (AI can only create/modify here)

│   ├── CHANGELOG.md      # Auto-generated changelog (AI may only append)

│   └── applied\_patches.txt # Record of applied patches (managed by CI)

├── .github/

│   └── workflows/

│       └── build.yml     # CI pipeline (AI must NEVER modify)

├── docs/                 # Recommended: modular documentation (CI auto-assembles into README.md)

├── CHANGELOG.md

└── README.md             # Never edit directly (generated by CI from docs/)

**CI must support** (verified build.yml workflow):

- Detect new ZIPs in zip\_update/patches/

- Automatically extract and apply patches (supports apply.sh scripts)

- Update applied\_patches.txt and CHANGELOG.md

- Build the project (firmware / APK / Docker, etc.)

- Create GitHub Release + Git Tag

- Honor #!build / #!no-build / \[skip ci] markers

---

## 3. AI Three-Step Workflow (Strictly Required)

Every AI response **must** follow these three steps exactly. Never deliver code in one step.

### Step 1: Proposal

- List all files to be created/modified/deleted (full paths relative to repo root)

- Explain the purpose of each change

- Ask whether you need the current source files first

### Step 2: Plan

- Provide a precise change plan for every file

- Describe impact on existing architecture

- Confirm compliance with file-size limits

### Step 3: Code Delivery

- Output **only** the files that need to change (never full files unless they are brand new)

- Package everything into a **rule-compliant ZIP patch** (see Section 4)

- Suggest a clear patch filename (e.g. update\_0422\_feature\_xxx.zip)

**AI must request current source files** before generating any code. Never rely on outdated memory.

---

## 4. Hard Patch Delivery Rules (AI must obey 100%)

1. **ZIP Naming**:

- Recommended: update\_YYYYMMDD\_feature\_description.zip

- Alternative: sequential update\_001\_fix\_xxx.zip

2. **ZIP Content Requirements**:

- Contain **only** files that actually changed

- Paths must be relative to repository root (e.g. src/main/kotlin/com/example/App.kt)

- May include an apply.sh script for complex migrations (CI runs it first)

- **Never** include .github/, .gitignore, CI config files, or any infrastructure

3. **File Size Limits**:

- After modification, each file should be **≤ 300 lines** (strongly prefer ≤ 200 lines)

4. **CHANGELOG Update**:

- Add a summary as the **first line** of zip\_update/CHANGELOG.md

- Example:

```

#!build

2026-04-22 feat: Add course export feature (by AI)

- New ExportViewModel.kt

- Modified CourseRepository.kt

```

5. **Strictly Forbidden** (AI must never do):

- Edit README.md directly (use docs/ instead)

- Touch any file under .github/workflows/

- Modify root build files (build.gradle, platformio.ini, etc.) unless explicitly authorized

- Include binary files in patches (except required assets)

---

## 5. General Code & Architecture Requirements

- **Single Responsibility**: One file = one clear purpose

- **Modular design**: Prefer Manager / Service / Repository / ViewModel patterns

- **State management**: Use StateFlow / LiveData / Compose State where applicable

- **Documentation**: Add a brief comment header to every new or modified file

- **Testing**: Consider unit tests for new features (recommended)

- **Compatibility**: Changes must pass CI build

---

## 6. Documentation Rules

- All documentation lives in the docs/ directory (split by feature)

- docs/README\_\*.md files are automatically merged into root README.md by CI

- AI only edits files inside docs/ — **never** the root README.md

## CI_BUILD_PIPELINE_RULES.md

**GitHub Actions `build.yml` Rules (Universal CI Pipeline)**  
**Version**: 1.0  
**Purpose**: Standardize the patch-based CI workflow for any project (firmware, Android APK, web app, etc.).  
**Core Idea**: When an `update_*.zip` appears in `zip_update/patches/`, the pipeline automatically applies it, commits the result, builds the project, and creates a release — all without human intervention.

---

### 1. Core Principles (Same as AI Rules)

- AI **never** touches `.github/workflows/build.yml`
- All patch handling and building is 100% automated
- Every patch application is logged and committed with `[skip ci]`
- Human retains final control (review + manual push if needed)

---

### 2. Required Trigger Conditions

```yaml
on:
  push:
    branches: [ '**' ]
  pull_request:
    branches: [ main ]
  workflow_dispatch:
    inputs:
      force_build:
        description: 'Force build even if no changes'
        required: false
        default: false
        type: boolean
```
### 3. Job Structure (Mandatory Order)
The workflow must contain these three jobs (in this dependency order):
Job 1: check (Pre-flight Check)

Runs on every trigger
Finds the newest update_*.zip in zip_update/patches/ (sorted numerically by filename)
Outputs:
patch_found
patch_path
should_apply
compile_xxx (project-specific build flag)


Additional recommended checks (from APK version):

Detect relevant file changes (app/, zip_update/, docs/)
Check if patch was already applied (zip_update/applied_patches.txt)
Read build flag from first line of zip_update/CHANGELOG.md:


\# !build → compile


\# !no-build → only apply patch, skip compilation


Job 2: apply-patch (Only if patch exists)

Runs only when should_apply == true
Security validation: Abort if patch contains .github/ directory
Extract ZIP → rsync into repository root
If zip_update/patches/apply.sh exists → run it (chmod +x first)
Cleanup:
Delete the consumed *.zip and apply.sh
Append patch name to zip_update/applied_patches.txt (keep last 50 entries)

Commit:Bashgit commit -m "chore: apply patch ${PATCH_NAME} [skip ci] $(head -1 zip_update/CHANGELOG.md)"
git push
Uses GITHUB_TOKEN with contents: write

Job 3: build (Project-specific compilation)

Runs when:
compile_xxx == true
apply-patch succeeded or was skipped

Uses heavy caching for speed:
pip / Gradle
PlatformIO toolchain & libraries (if applicable)
Project-specific cache keys (e.g. platformio.ini, build.gradle)

After successful build:
Create GitHub Release
Upload artifact (.bin, .apk, etc.)
Auto-generate tag (e.g. v1.076)
